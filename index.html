Dylan Dog Citation LODLAM Project

A project to create a linked open data (LOD) representation of Dylan Dog inspirations and citations.

By Alessandro Rocchi

Domain Analysis

Project Idea

The main idea behind this project is to create a linked open data (LOD) representation of Dylan Dog inspirations and citations. The goal is to provide a structured and semantically rich dataset after the extraction and linking of information.

The first step in this process is to identify and extract relevant information from the Wikipedia page. I chose to focus on the Italian Wikipedia page Dylan Dog, which refers to the fictional character created by Tiziano Sclavi.

This page was chosen instead of the one related to the comic series because it provides a more focused view on the character and an exaustive section for citations and references.

Entities

After choosing the page, I identified a total of 27 entities that were mentioned in the Italian Wikipedia page. Every entity has been linked to their authority control where applicable. The main authority control used was VIAF

Knowledge organization: elaborate models

Theoretical Model

After identifying the key entities and their relationships, I developed a theoretical model to represent the knowledge structure of Dylan Dog citations. This model aims to capture the semantic connections between different types of entities, and to show connections which weren't explicitly mentioned in the original text.

The model is composed of three main components: the main entities, their attributes, and the external resources (URIs) used to link them to authoritative sources or catalogues. In particular, the autoritative sources used are: VIAF and OPAC SBN. Wikidata has been used as a semantic disambiguation for entities which are not bibliographic. For the location Geonames has been used as the authoritative source.

The graph was created using yED software and a zoom JavaScript plugin was included to enable it's interactive exploration and to make it more readable. On the right you can find a legend explaining the symbols used in the graph.

Why creating a new class instead of using an existing one?

In my project I created the class dd:Albo instead of using the existing cbo:Issue because in the Italian comic industry the term "albo" refers to a specific type of publication that has unique characteristics not covered by the "issue" concept which is more related to the American comic industry. In particular an "albo" in Italy is a monthly publication that contains a self-contained story. Sometimes the story includes some references to a horizontal plot, but this is not mandatory.

On the other hand, an American “issue” refers to a specific installment in a series that will later be compiled into a single volume to form a complete story. In fact, in the U.S., there is the concept of a “run”: a series of installments that must be read in order to fully appreciate the entire plot. In contrast, in the traditional Italian comic book scene (with a few exceptions), this concept does not exist, and the "albi" are self-contained.

So, to better describe my domain I created the new class dd:Albo which is a specialized entity meant to represent the unique characteristics of Italian comic publications. I decided to make it a subclass of the existing cbo:Issue class because it shares some common properties with the existing class, but it also has distinct characteristics that warrant its own classification.

Why creating a new set of Object Properties?

I created a new set of Object Properties to better capture the relation of Inspired By between different entities in my domain.

In particular, I created the following Object Properties:dd:inspiredBy which represents the relationship between an entity and the entity that inspired it, this property is a sub-property of the more general prov:wasInfluencedBy property.

I then created the properties: dd:appearanceInspiredBy, dd:familyNameInspiredBy, dd:givenNameInspiredBy, to describe the relationships between the various entities in the domain in even greater detail. All of these properties are subproperties of dd:inspiredBy to maintain the logical and semantic consistency of the domain.

In addition, I also created the property dd:isATributeTo and its subproperty dd:isNameATributeTo. These properties are necessary to more accurately represent tributes, which differ semantically from inspirations, to people or characters.

Real-world Entities in the XML/TEI file

In the XML/TEI file, I annotated the following real-world entities:

  • People (authors, actors, etc.)
  • Places (locations)
  • Organizations (publishers, production companies)
  • Works (comic, books, movies)

All of them were annotated with relevant metadata, a unique identifier, and reconciliated with authority files, in particular I use VIAF and OPAC. In this section you can find an example for each entity type.

People

                                            <persName>
                                                <surname>Marx</surname>
                                                <forename type="middle">Henry</forename>
                                                <forename type="first">Julius</forename>
                                            </persName>
                                            <persName type="pseudonym">
                                                <surname>Marx</surname>
                                                <forename type="first">Groucho</forename>
                                            </persName>
                                        </person>

Here we can see an example of a person entity annotated in the XML/TEI file. The entity represents Groucho Marx, an actor, and includes his full name, pseudonym, and a link to his VIAF authority record for reconciliation. I used the @role attribute to specify that Groucho Marx is an actor in the context of the Dylan Dog Wikipedia page and the @sameAs attribute to link the entity to its corresponding authority record in VIAF.

Since the entity has a pseudonym, I chose to indicate this by using another <persName> tag and assigning it the attribute @type="pseudonym" to specify that the content is the pseudonym and not the legal name.

In this case, there isn't a single <author> tag because a comic is a complex entity that can be created through the work of multiple people, each with their own specific role in the production process. Therefore, I decided to divide the various roles into responsibilities in order to better represent the resource. I used the same approach for movies as well.

Relations

                                        <relation active="#dd_nr1" name="cbo:writer" passive="#t_sclavi"/>
                                        <relation active="#dd_nr1" name="cbo:artist" passive="#a_stano"/>
                                        <relation active="#dd_nr1" name="cbo:coverArtist" passive="#c_villa"/>

In my XML/TEI file, I also wanted to explicitly specify the relationships between the various annotated entities and indicate which domain class they belonged to. This approach helped when generating the dataset using a Python script, since all that was needed was to translate the existing triples.

The relationship is defined using the <relation> tag, where @active is the subject, @name is the predicate, and @passive is the object.

Conclusion

In conclusion, this project has provided a comprehensive exploration of the domain of Dylan Dog, utilizing various ontologies and data representation techniques to create a rich and interconnected dataset. The use of XML/TEI for initial data annotation, followed by transformation into HTML and RDF formats, has allowed for effective data management.

The creation of custom classes and object properties has enabled a more accurate representation of the unique characteristics of Italian comic publications, while the integration of external authority files has ensured the reliability and verifiability of the data. Overall, this project help me to understand the power of semantic technologies in organizing and representing complex information in a meaningful way.
